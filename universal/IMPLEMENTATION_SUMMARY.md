# Полное описание выполненных работ по проекту Universal

## Контекст и цель
В ходе работы мы последовательно реализовали и доработали ключевые механики MVP для сущностей и модулей корабля в соответствии с архитектурными принципами проекта:
- Event Bus через `GameEvents`
- Feature-based folders
- Composition over Inheritance
- Data-driven подход через экспортируемые параметры

Основной фокус: мусор (`Debris`), его спавн, автоматический сбор (`Collector`), модульная база, ядро корабля и правила строительства/энергии.

---

## Этап 1. Базовый мусор Debris

### Что было сделано
1. Создан компонент кликабельности `ClickableComponent`.
2. Создана сцена `Debris` и скрипт `debris.gd`.
3. Подключены 3 варианта визуала мусора:
- `trash-1.png`
- `trash-2.png`
- `trash-3.png`
4. Введены разные награды металлом в зависимости от типа мусора.
5. Добавлены явные настраиваемые параметры в инспекторе для:
- награды за каждый тип мусора
- скорости движения
- размера юнита
- направления движения
6. Размер мусора зафиксирован как 1 универсальная единица (90x90).
7. Реализованы два способа сбора:
- ручной клик
- программный автосбор (для будущих сборщиков)
8. Сбор идёт через Event Bus: `GameEvents.garbage_clicked.emit(amount)`.

### Важные технические решения
- Добавлена защита от повторного начисления перед `queue_free()`.
- Движение и баланс вынесены в экспортируемые переменные.
- Добавлен API для будущей автоматизации (`auto_collect`, проверка радиуса).

## Этап 2. Спавнер мусора

### Требование
Мусор появляется сверху за границей видимой области, летит вниз, появляется в кадре.
Раз в 2 секунды с вероятностью 50%.
Максимум 20 объектов на экране.

### Что реализовано
1. Добавлен `debris_spawner.gd`.
2. Логика спавна по таймеру:
- `spawn_interval_sec = 2.0`
- `spawn_chance = 0.5`
3. Лимит активного мусора:
- `max_debris_on_screen = 20`
4. Спавн происходит выше экрана (`y < 0`) и движение вниз.
5. Добавлена очистка мусора после ухода ниже экрана.
6. Спавнер подключен в `game_board.tscn` как дочерний узел.

### Результат
Поток мусора стабилен, управляем параметрами и соответствует заданному сценарию появления.

## Этап 4. Логика ядра и обороны

### Что реализовано в `CoreModule`
1. Ядро имеет размер 2x1.
2. Инкапсулирована статка `Defence`.
3. Добавлен метод разрешения столкновений по механике:
- сравнение `Defence` и `hazard_class`
- при провале: расчёт потерь модулей `(hazard - defence) / 10`
- при успехе: скидка 50% на ограниченное число построек `hazard / 10`
4. Выброс событий о смене защиты и результате столкновения через `GameEvents`.

### Примечание
Добавлен фундамент логики столкновений, пригодный для будущей полной боевой фазы.

## Этап 5. GridManager и правила строительства

### Что реализовано
`grid_manager.gd` расширен до полноценного валидатора строительства:
1. Учёт занятых ячеек.
2. Учёт «энергизированных» ячеек.
3. Метод `canBuildAt(pos, module_type, size) -> bool`.
4. Питание от ядра в радиусе 1 клетки.
5. Питание от реактора в радиусе 1 клетки (без привязки к стороне).
6. Запрет ставить реактор в зоне действия уже стоящего реактора.

### Важный момент
`GameBoard` не дублирует правила и использует только `gridTileManager.canBuildAt(...)` для принятия решения.

## Этап 6. Интеграция в GameBoard и UI

### GameBoard
1. Убран старый временный рендер корабля через `ColorRect`.
2. Добавлен управляемый root для модулей.
3. Создаётся и регистрируется Core при старте.
4. Обработка `GameEvents.build_requested`:
- выбор позиции
- проверка `canBuildAt`
- списание металла
- размещение модуля
- `module_built.emit(...)`
5. Обновляются границы корабля (используются в логике сборщика).
6. Название переменной `gridTileManager` использовано в явном виде.

### UI
1. Обновлены кнопки сборки:
- Reactor (350 metal)
- Collector (100 metal)
2. Кнопки эмитят build-запросы через `GameEvents` с корректными ID из `Constants`.

## Этап 7. Доработка логики сборщика (финальная версия)

### Новое требование
- Автоматический сбор в радиусе **5 клеток от края корабля**.
- После срабатывания сборщик уходит в кулдаун **5 секунд**.
- Переосмыслить предыдущую логику под это правило.

### Что изменено
1. `collector_module.gd` переписан:
- `collect_radius_from_ship_edge_cells = 5.0`
- `collect_cooldown_sec = 5.0`
2. Логика больше не опирается на «полусферу по направлению модуля».
3. Цель выбирается по минимальной дистанции до границы прямоугольника корабля.
4. Мусор внутри корабля игнорируется.
5. Сохраняется визуальный лазер до выбранной цели.
6. Цикл работы:
- попытка сбора
- запуск таймера кулдауна 5 сек
- новая попытка после кулдауна

### Принятое поведение
Кулдаун стартует после каждого цикла проверки (даже если мусор не найден). Это отражено в текущей реализации.

## Архитектурные принципы, которые соблюдены
1. Нет прямых связей с UI/ресурсами из сущностей мусора и модулей.
2. Начисление металла идёт через Event Bus (`garbage_clicked`).
3. Логика разбита на переиспользуемые компоненты и независимые модули.
4. Ключевые параметры сделаны экспортируемыми и явными.

## Итог
На данный момент в проекте есть рабочий вертикальный срез геймплейного цикла:
1. Мусор спавнится сверху и движется вниз.
2. Мусор можно собирать кликом и автоматически через сборщики.
3. Есть модульная база для расширения типов модулей.
4. Ядро 2x1, реактор и сборщик интегрированы в общую систему строительства.
5. Применены правила энергии и ограничения установки через `canBuildAt`.
6. Логика обороны ядра подготовлена под дальнейшую боевую механику.

Этот фундамент подходит для следующего шага: ручной режим выбора клетки для строительства, визуальный предпросмотр установки и перенос балансовых параметров модулей в `.tres`.
